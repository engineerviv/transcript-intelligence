# Transcript Intelligence Platform

AI-powered business intelligence extraction from enterprise call transcripts.

## Setup

**Prerequisites:** Python 3.11+, an OpenAI API key.

```bash
pip install -r requirements.txt
```

Create a `.env` file in this directory:

```
OPENAI_API_KEY=sk-...
```

## Running

**Step 1 — Run the pipeline** (processes transcripts, calls the LLM, writes outputs):

```bash
python run_pipeline.py
```

This is safe to re-run. The LLM enrichment step is cached by prompt hash (`outputs/cache.json`), so only new or changed transcripts cost API calls. The data loading step is also skipped if `data/transcripts.json` already exists.

**Step 2 — Launch the dashboard:**

```bash
streamlit run app/streamlit_app.py
```

Opens at `http://localhost:8501`.

## Project Structure

```
transcript-intelligence/
├── run_pipeline.py        # Entry point — runs all three pipeline steps
├── src/
│   ├── data_gen.py        # Loads raw dataset, infers call types, writes transcripts.json
│   ├── llm.py             # OpenAI wrapper with JSON mode, caching, retry
│   ├── pipeline.py        # LLM enrichment — extracts topic, sentiment, urgency, churn risk, etc.
│   ├── analysis.py        # Statistical aggregation + LLM-generated executive insights
│   ├── chatbot.py         # RAG-style Q&A over the transcript dataset
│   └── utils.py           # Shared helpers (colors, badges, load_json)
├── app/
│   └── streamlit_app.py   # Interactive dashboard (4 tabs)
├── data/
│   └── transcripts.json   # Normalized transcripts (generated by data_gen)
└── outputs/
    ├── enriched.json      # LLM-enriched transcripts (generated by pipeline)
    ├── aggregated.json    # Aggregated stats + executive insights (generated by analysis)
    └── cache.json         # LLM response cache (keyed on prompt hash)
```

## Design Decisions

**Why LLM-based categorization over clustering?**
The dataset is ~100 transcripts. Statistical clustering (k-means, LDA) is noisy at this scale and produces labels that require manual interpretation anyway. GPT-4o-mini extracts business-meaningful labels (topic, intent, churn risk) directly from the summary text, grounded in domain context. It runs in under 2 minutes and costs less than $0.05 for the full dataset.

**Why use the summary instead of the full transcript for enrichment?**
Each full transcript is ~5–15k tokens. The pre-extracted `summary.json` (from the dataset provider) captures the substance in ~200 tokens. Using the summary reduces cost by ~10x and avoids context-window noise with no meaningful quality loss for the BI extraction task.

**Why file-based caching?**
The enrichment prompt is deterministic (temperature=0). Caching by SHA-256 hash of the prompt means re-running the pipeline never re-calls the API for transcripts already processed. This makes iterating on the aggregation and dashboard steps free.

**Why no vector database for the chatbot?**
At 100 transcripts, serializing the top 20 most relevant records directly into the prompt is faster, simpler, and cheaper than maintaining embeddings. The chatbot uses keyword-based relevance scoring to select which transcripts to include, with urgency/sentiment as a tiebreaker. A production version with thousands of transcripts would swap this for a semantic search layer.

**Call type inference**
The raw dataset has no explicit `call_type` field. Types are inferred from meeting titles:
- `"Support Case #..."` → `support`
- `"Aegis / <Company> ..."` → `external`
- Everything else → `internal`
